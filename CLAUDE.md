# Quay（包名 dbmcp / CLI dbm）

**品牌名 Quay（码头）**：本地数据库工作台——查询台 · Redis 控制台 · 分析工作台 · MCP（给 agent）四个前端共享一套连接与安全治理主线。改名只是对外品牌：**包名仍 `dbmcp`、CLI 仍 `dbm`、配置目录 `~/.config/db-manage-mcp`、launchd label `com.db-manage-mcp`、keyring service `db-manage-mcp` 全部不变**（只换品牌、不做全量重命名，避免动迁移逻辑）。按项目管理连接与账密、SSH 多层跳板、SQL 审计与人工授权、操作审计与管理后台。完整设计见 `DESIGN.md`（改动核心流程前先读它）。

## 技术栈（已确认，勿擅自更换）

- Python 3.12 + FastMCP（官方 MCP SDK）+ FastAPI（MCP 与管理后台共用一个 ASGI 应用）
- sqlglot 做 SQL AST 解析与审计；SQLAlchemy Core（不用 ORM）+ redis-py
- SQLite 存审计/审批单/元数据缓存；管理后台服务端渲染 HTML（未用 Jinja2/HTMX）
- SSH 多跳复用系统 OpenSSH（`ssh -J jump1,jump2`），不自己实现隧道协议
- 部署形态：**本地进程模式**（`uv run dbm serve`），macOS 用 launchd 常驻（scripts/install-launchd.sh）；stdio 作单 agent 直连模式。**已弃用 Docker**——本地单机场景 Docker 只带麻烦（连宿主库绕网络、无 keyring 后端、SSH key 变容器内路径）

## 安全设计红线（实现时必须遵守）

1. **默认拒绝**：SQL 解析失败、多语句、无法分类的一律按写操作处理，进审批流。
2. **密钥不落明文**：配置文件只存 `env://` / `keyring://` 引用；密码永不出现在日志、审计记录、工具返回值中。本地进程模式下管理后台可把页面输入的密码写 keyring；`env://` 引用的值由常驻服务从 `~/.config/db-manage-mcp/env`（600）注入。
3. **拒绝—重提 + change_id 放行**：写操作先被明确拒绝并生成审批单（含风险报告），人在后台批准后 agent 带 change_id 重提才放行；**执行的永远是审批单里存储的 SQL**，重提文本只作指纹校验（不一致即拒），指纹匹配仅作无 id 时的兜底。审批单一次性核销、有 TTL（30 分钟）。prod 环境强制审批。
4. **双账号**：日常查询走只读账号，仅审批通过的执行切换 writer 账号。
5. **连接与密钥管理不暴露为 MCP 工具**（agent 碰不到）；人可通过 CLI 或**已登录的管理后台**操作。后台需认证（DBM_ADMIN_TOKEN），页面输入的密码一律进 keyring、配置只存引用。
6. 通知遵循"安静即正常"：不主动推送，审批挂起由 agent 在会话中告知用户。

## 约定

- 每条 SQL 必须落审计记录：agent（clientInfo + session）、时间、连接、SQL 原文与指纹、风险等级、结果（行数/耗时/状态）、审批信息。
- 查询默认加 LIMIT（1000）与语句超时（30s），可按连接配置。
- Redis 走命令分类模型：读命令直通；KEYS / FLUSHDB / FLUSHALL / CONFIG 属 CRITICAL 需审批。

## 经验教训

- **包名是 `dbmcp`，不是 `dbm`**：`dbm` 是 Python 标准库模块，会被 stdlib 遮蔽导致 import 全挂。CLI 命令仍叫 `dbm`。
- **本机测本地服务要绕过代理**：这台机器 shell 有 SOCKS 代理环境变量，不绕过会把 127.0.0.1 请求发进代理得到 502。curl 用 `--noproxy '*'`；fastmcp Client（底层 httpx）用 `env NO_PROXY='*' no_proxy='*'`（单纯 `-u ALL_PROXY` 不够，httpx 还会读其他代理变量，NO_PROXY 最稳）。
- **写 sqlglot 相关逻辑前先跑实验脚本验证解析行为**：已验证的坑——PG 方言 `EXPLAIN`/`SHOW` 退化为不透明 `Command` 节点需特判；`WITH x AS (INSERT ...) SELECT` 顶层是 `Select`，必须遍历整棵树找写节点。
- fastmcp 3.x 装的是 `fastmcp` + `fastmcp-slim`，其依赖 `uncalled-for` 是正常的依赖注入库（已核实非投毒包）。
- **MySQL 会话设置的多条语句不能逗号拼接**：`SET SESSION max_execution_time=X, SESSION TRANSACTION READ ONLY` 是非法语法（1064）——`SET TRANSACTION READ ONLY` 是独立语句、不是变量赋值。必须用 connect 事件监听器逐条 execute。此 bug 只有真实 MySQL e2e 才暴露，SQLite 单测发现不了；已抽 `mysql_session_statements()` 纯函数 + 回归测试（test_engines.py）。**教训：DB 方言相关代码，SQLite 单测不够，必须对目标 DB 跑真实 e2e。**
- **大 DELETE/UPDATE 报 2013「Lost connection ... read operation timed out」的根因与修法**：MySQL 的 `max_execution_time`（`mysql_session_statements` 设的）**只约束 SELECT，不约束 DELETE/UPDATE/DDL**——所以长写操作没有服务端语句超时，只被 pymysql 的 **socket `read_timeout`（原来 reader/writer 都 = `statement_timeout_s`=30s）** 打断成 2013。修法：**按角色分超时**（`engines.role_timeouts`）——reader 的 socket 超时用 `statement_timeout_s`，**writer 用新增的 `policy.write_timeout_s`（默认 600s，连接可配）**；PG 的 `statement_timeout` 对所有语句生效，writer 也换用 `write_timeout_s`。放大写超时让正常大写跑完，跑飞的写用**查询台取消按钮 KILL**（见下条）。真实 MySQL e2e：`scripts/e2e_serial_cancel.py`（`DO SLEEP(n)` 模拟长写——DO 非 SELECT 故 max_execution_time 不管它，忠实复现 DELETE 场景）。
- **查询台 SQL「忙时拒绝」+ 计时 + 取消（`jobs.py::JobManager`）**：后台异步查询原本每个 job 起一个 daemon 线程、无序并发。**同一 key（=连接）一次只跑一条 SQL，忙时新提交直接 `raise Busy` 被拒**（一开始做过 FIFO 排队，但排队位置对使用者不清晰，改成明确拒绝——「该连接有查询正在执行，请等待或取消后再试」）；不同连接、workflow/画布（`object()` 唯一 key）各自并行、不互拒。前端 `run()` 另加同 tab 守卫（本 tab 在跑就 flash 拒绝，避免同 tab 重入的竞态）。**取消**：调用 `engines.make_canceller`——MySQL `KILL QUERY <thread_id>`、PG `pg_cancel_backend(pid)`、SQLite `interrupt()`，在**另开连接**上发（SQLite 例外，对同一连接 interrupt）；取消器经 `run_query/run_write` 的 `on_start` 回调注册。计时用**客户端秒表锚定服务端 `elapsed_ms`**（`jobRunAt = Date.now()-elapsed_ms`，平滑且刷新后续接准确，别纯靠轮询值会显得偏快）。执行状态图标用 **Monaco 字形装饰画在被执行语句起始行**（`execLineFor` 要跳过语句区间起点的前导空白/换行，否则错位到上一行），不再固定在编辑器第一行。**出错/取消也进结果 tab**（红色「✗ 结果 N」页），与成功结果统一；**关闭结果 tab 只切换展示、绝不触发重查**（旧 bug：`closeResult`→`selectResult` 对已释放数据的邻页自动重跑）。真实 MySQL e2e + 浏览器 e2e 全过。
- 真实 MySQL 9.5 e2e 已验证通过：reader 只读防线报 1792、writer 双账号写入、information_schema 行数估算、索引命中风险判定、完整拒绝—重提审批闭环、管理后台批准。
- **防「大表拉挂 DB」用注入 LIMIT，别用流式游标**：pymysql 默认缓冲游标会把整个结果集读进客户端内存，`SELECT * FROM 大表` 无 LIMIT 时把进程/DB 拖挂。试过 `stream_results=True`（SSCursor）——但提前 fetchmany 后关连接、连接归还池复用会报 `Previous unbuffered result was left incomplete` / commands out of sync。**正解：给缺 LIMIT 的顶层 SELECT/UNION 注入 `LIMIT max_rows+1`（`engines.paginate_sql`，sqlglot AST，用户自带 LIMIT/非 SELECT 不动），让 DB 只返回这么多行，缓冲游标即安全、池复用干净**。查询台在此基础上做 LIMIT/OFFSET 分页（取 page_size+1 探测下一页，走 `_read` 不受连接 max_rows 二次截断）。
- **未绑定 database 的 MySQL 连接不能直接反射**：SQLAlchemy `inspect(engine).get_table_names(schema=None)` 在无默认库时，取 `SELECT DATABASE()` 得 None，`_escape_identifier(None)` → `None.replace(...)` 崩（报 `'NoneType' object has no attribute 'replace'`）。修法：查询台对未绑库的 MySQL/PG **先列库**（`engines.list_databases` = `inspect().get_schema_names()` 过滤系统库），选库后 `get_table_names(schema=db)` / `describe_table(table, schema=db)` 带 schema 反射；`service.list_tables(schema=None)` 对无库连接直接抛清晰错误而非让它崩。这也是 DataGrip 的库→表→列三级树。
- **子进程活着时别 read() 它的管道**：SSHTunnel 隧道就绪超时路径先 `stderr.read()` 再 close——ssh 还活着时 read() 阻塞到进程自行退出（DNS 挂起时数分钟），把探测/全量测试整个卡死；且该 bug 被网络环境掩盖（ssh 秒败走"提前退出"分支不触发），只在 DNS 变慢的日子爆发。正解：**先 terminate/kill 子进程再 read**（进程死后 read 立即 EOF 且保留已缓冲内容）。排查特征：进程 CPU 时间停住不涨、wall clock 在走 = 阻塞在外部 IO。
- **后台任务线程必须 daemon**：admin 异步查询最初用 `ThreadPoolExecutor`——它的工作线程**非 daemon**，每个 test_admin 用例 mount_admin 都建一个池，pytest 用例全过却在退出时挂死（表现为"全量测试跑不完"，极难定位）。改为 `threading.Thread(daemon=True)` 逐任务起线程。凡常驻服务里的辅助线程（housekeeping、job worker）一律 daemon。
- **Monaco 无构建 vendoring 的坑**：① Monaco 的 AMD loader 运行时按路径拉几十个文件，静态路由必须支持子路径（`{path:path}`）；② web worker 用 data-URI `importScripts(workerMain.js)` 加载，若静态路由带 `@guard`，worker 请求带不上 cookie 会被 303 到登录页而崩——所以 `/admin/static/*` **不加鉴权**（都是公共库文件、无敏感数据）；③ 用 `automaticLayout:true` 让 Monaco 自适应容器尺寸，省去手动 layout。装法：`npm registry` 取 `monaco-editor` tgz 解 `package/min/vs` 到 `static/monaco/vs`。
- **前端全屏页别用 `position:absolute;inset:0` 铺满**：会相对视口定位、盖住后台左侧导航栏。用 `height:100vh` 让它待在 `main` 网格列内即可。
- **自家静态文件（console.js/css）必须 `Cache-Control: no-cache`**：曾设 max-age=86400，迭代时浏览器一直跑旧 JS，连坑三次（新功能"没生效"、旧 bug"复现"，其实全是缓存）。现在静态路由区分：vendor（monaco/、vue.*，内容不变）长缓存，其余 no-cache（每次带 If-None-Match 校验，改完即生效，无需硬刷新）。
- **合成打字对代码编辑器（CodeMirror/Monaco）不可靠**：浏览器自动化 `type` 会掉字符；e2e 验证时用 `javascript_tool` 直接 `monaco.editor.getModels()[0].setValue(...)` 或 CM 的 `setValue`。
- **DuckDB 的 `DROP TABLE IF EXISTS` 遇同名 VIEW 报类型错**（IF EXISTS 只豁免"不存在"，不豁免"类型不符"）：删数据集必须先查 information_schema 拿 table_type 再按类型 DROP（analysis.drop_dataset）。
- **MySQL DECIMAL 快照进分析工作区会变字符串**：`engines._jsonable` 把 Decimal 转 str，DuckDB 导入推断成 VARCHAR，`sum()/avg()` 报 Binder Error。修在 `analysis._infer_types`：某列非空值全是数字字符串 → 推断 DOUBLE（回归测试 test_decimal_strings_become_double）。真实暴露路径：DAG 聚合节点对 amount 求和。
- **UMD 库必须放在 Monaco loader.js 之前加载**：loader.js 定义了全局 `define.amd`，其后加载的 UMD 包（如 echarts）会走 AMD 注册而**不挂 window.xxx**，表现为"脚本 200 加载成功但全局变量 undefined"。修法：`<script>` 顺序上把 UMD vendor 放到 loader.js 前面（见 admin.py `_console_body`）。
- **sqlglot 的 TokenizeError 不是 ParseError**：引号不闭合等词法错误在**分词阶段**抛 `TokenizeError`，不是 `ParseError`；二者都继承 `sqlglot.errors.SqlglotError`。`classify`/`assess` 只 catch `ParseError` 会让非法 SQL 直接抛异常冒泡（agent 的 execute 走 server.py 只 catch QueryRejected/KeyError/ValueError → 变 500；违反「默认拒绝」红线）。**必须 catch 基类 `SqlglotError`**，任何解析/分词失败都默认拒绝按写操作/最高风险处理。回归测试见 test_classify.py::test_tokenize_error_rejected_not_raised、test_risk.py::test_tokenize_error_is_critical_not_raised。
- **Vue 模板只能访问 data()/computed/methods 声明的属性**：命令式 `this._docCmd = x` 赋的属性**不是响应式、模板里也取不到**（`{{ _docCmd }}` 直接 ReferenceError 让整个组件 render 崩、白屏）。要在模板显示的中间状态必须在 `data()` 里声明。排查白屏第一步永远是看浏览器 console 报错（这次连踩两个：`countLeaves 未定义`的模块级辅助函数漏写、`_docCmd`未声明——都表现为整页白屏只剩侧栏）。
- **Redis 控制台单开一页而非融进查询台**：查询台是深度面向 SQL 关系模型的（分页/WHERE/列头排序/单元格编辑/图表/DAG/schema 选择器…），Redis 是键值模型，硬塞会到处 `isRedis` 条件分支、回归风险高、两边互相牵制。独立页 `/admin/redis`（`redis.js`/`redis.css`）资产级复用 console.css 外壳 + `--dg-*` 变量 + `.dg-btn`，两个工具各自简单。**教训：两种数据模型差异大时，单开一页的少量基础设施重复，远比塞进一个控制台的隐性复杂度便宜。**
- **ProxyJump 的两个坑（多跳 e2e 踩出）**：① 命令行 `-o` 选项只作用于最终目标、不传递给跳板连接——自定义密钥/known_hosts 必须用 `-F 配置文件`（对链上每跳生效），对应 `ssh_options: ["-F", path]`；② sshd_config 是**首个匹配生效**，往文件末尾 append `AllowTcpForwarding yes` 无效，容器里用 `sshd -o` 命令行覆盖。多跳验证脚本：`scripts/e2e_ssh_multihop.sh`（两级 sshd 容器链到本地 MySQL，凭证全程隔离在 /tmp）。
- **每跳独立证书 = 生成 `ssh -F` 配置，不能靠命令行 `-i`**：`jump_hosts` 是结构化 `list[JumpHost]`（host/user/port/`identity`引用证书库或内联`key_path`/`known_hosts_path`），旧的裸字符串 `"user@host:port"` 由 `JumpHost` 的 before-validator（`parse_hostspec`）兼容解析——所以老 YAML/e2e 全不用改。隧道层（`tunnel.py`）分两条路：**无任一跳带 key** 走遗留 `-J`（不生成临时文件、行为不变，保护现有 e2e）；**任一跳带 key** 走 `build_ssh_config` 生成临时 ssh 配置（`Host *` 放 BatchMode 等全局项对每跳生效，每跳 `Host dbmhopN` 带自己的 IdentityFile/UserKnownHostsFile + `ProxyJump` 串链），`ssh -F <tmp>`，`close()` 时 `os.unlink`。证书库存 YAML 顶层 `ssh_identities`（只存路径引用、不存密钥内容），引擎池/Redis 池各持 `AppConfig.ssh_identities` 的**活引用**（`self.pool.identities = self.config.ssh_identities`，同一 dict 原地增删即时可见，避免 14 处 `pool.get` 逐个透传）。删证书时被连接引用则拒删（`identity_referers`）。**每跳不同证书只能真实 e2e 证明**：`scripts/e2e_ssh_multihop_percert.sh`（jump1 只认密钥A、jump2 只认密钥B，正向查通 + 负向把 jump2 换密钥A 必须失败——单测测不出证书是否真被分别使用）。后台连接表单跳板改**可增删的行**（每行选证书或内联 key，纯 JS clone `<template>`），证书库在系统设置「SSH 证书」tab（`?tab=ssh`）。

- **写语句的 EXPLAIN 必须用 writer 账号**：`admin_explain` 原来固定用默认 reader（只读账号）取执行计划，对 `DELETE/UPDATE/INSERT` 会被 DB 以权限不足拒绝（MySQL `1142 command denied to 'agent_read'`）——EXPLAIN 虽不真正执行 DML，但服务端仍按语句所需权限校验。修法：`role = "writer" if (not verdict.readonly and cfg.writer is not None) else "reader"`（无独立 writer 才退回 reader，如 sqlite）。参照已有的 `_try_explain`（审批风险报告里的计划）本就是 reader→writer 回退。**sqlite 单测测不出**（writer 复用同库、无权限概念），靠真实 prod EXPLAIN + HTTP 级 spy 测试（`test_explain_write_uses_writer_role` 断言 DELETE 选 writer、SELECT 选 reader）证明。
- **实时 lint 对「空行分隔的多条 SQL」误报到下一条**：`sqlglot.parse` 只按**分号**拆多语句、不认空行分隔。所以编辑器里「第一条无分号 + 空行 + 第二条」被当成一条解析，在第二条起始处报语法错——红波浪线标到了没问题的下一条上（用户反馈「不明所以」）。修法（`admin._lint_sql`）：整体能解析就直接放行（避免拆块假阳性，如字符串内的空行）；整体报错时才按**空行**拆块（`_split_blank_line_blocks`，与前端 `stmtRanges` 一致）逐块 lint、把错误行号回填原文——合法块不报错、错误落到真正出错的那条。回归测试 test_admin.py::test_lint_blank_line_separated_statements。**注意**：`_split_blank_line_blocks` 是行级拆分（非引号感知），故用「整体能解析就放行」兜底假阳性；执行路径（stmtAtCursor/seqItems）用的是引号感知的前端 `stmtRanges`。
- **长 SELECT 报 2013「Lost connection」的根因与修法（reader 侧，与前面 writer 侧 2013 是姊妹坑）**：reader 的 socket `read_timeout` 原来 = `statement_timeout_s`（30s），而服务端 `max_execution_time` 也 = `statement_timeout_s`（30s）——**二者相等 → 竞态**：长 SELECT 到 30s 时，pymysql 的 socket read_timeout 常抢在 `max_execution_time` 把干净的 3024「超过最大执行时间」错误发回来之前触发，于是报成 2013「Lost connection ... read operation timed out」。修法：reader 的 socket read_timeout 加 `_READ_SOCKET_GRACE_S`（15s）宽限（`engines.mysql_read_timeout`），让服务端 `max_execution_time` 先触发、返回清晰错误；用户调大连接的读取超时（`statement_timeout_s`）时长 SELECT 也能真跑完而不被 socket 误杀。writer 侧无 max_execution_time 约束、socket=write_timeout_s 本身即上限、不加宽限。纯函数回归测试见 test_engines.py（socket read_timeout 必须 > max_execution_time）。
- **MySQL 9.x 的 EXPLAIN FORMAT=JSON 是新 schema（v2.0），字段与旧版完全不同**：旧版是 `{query_block:{table:{table_name,access_type:"ALL",rows_examined_per_scan,filtered,key,cost_info:{query_cost}}}}`；**新版**是 `{query,query_plan:{operation:"Table scan on orders",table_name,access_type:"table",estimated_rows,estimated_total_cost,inputs:[...]}}`——**新版 `access_type:"table"` 才是全表扫描（旧版是 `ALL`）**，行数在 `estimated_rows`、成本在 `estimated_total_cost`、还多了人类可读的 `operation` 串。查询台 EXPLAIN 概览 `walkPlan` 两种都认（`access_type` 小写 `all`||`table` → 全表扫描告警）。**教训：解析 EXPLAIN JSON 别只按旧 MySQL 文档写字段名，真机上先 dump 一次 tree 看实际结构**（这次概览首版按旧格式写，真机 demo-mysql 出来全是新格式字段、access 显示成 "table" 才发现）。
- **查询台多语句顺序执行的做法**（`console.js`）：选中多条一起执行 → `run()` 里 `splitStatements(sql)`（复用 `stmtRanges`，去尾分号）拆成单条，置 `t.seq={list,i}` 后 `run(false,0,parts[0])` 起第一条；带 `sqlOverride` 的递归调用不再进拆分分支。链式推进 `_seqAdvance(t,ok)` 挂在 `pollJob` 的每个终态出口：读/写成功 → `ok=true` 跑下一条，出错/取消/confirm（写需人工确认）→ `ok=false` 停整条序列。每条独立提交 run_async、客户端等前一条 job 终态再发下一条，服务端 (project,connection) 串行不会 Busy 互撞。**数据 tab（双击表名）不参与串行**：run_async 带 `parallel=1` → 服务端 `_solo_key()` 独立 key 并行，不占编辑器查询的连接名额。
